Design System In Action
引言
在数字时代，高响应力是必须的追求。那么谁挡在了提高响应力的路上，谁就必须要改变以获得更高的响应力。
当公司的业务开始扩张，我们要更快的构建更好的产品，以响应市场变化，这个时候，如何在各种场景下保证一致的用户体验，是如今设计师和前端工程师面临的挑战。
如今的设计师不仅仅是交付一套方案，更要交付一套系统，以保证后续诞生的所有方案都保持一致的用户体验，同时，前端工程师们还要保证这每一套方案都可以高速的实施，同时具有极高的质量。
巴别塔问题
前端工程师要处理的三个领域
设计领域
业务领域
技术领域
三个领域混杂在一起，极大的增加了信息的噪音，为高效交付软件设置了门槛
最终虽然会困扰前端工程师，但这三个领域只有靠业务人员、设计师和前端工程师一起协作，制定统一的语言，才能解决这个巴别塔问题
系统分类
尽量不要使用一个DS处理所有类型的系统，通过进一步划分系统类别，可以最大限度的降低DS的设计难度，实施提高成功率

典型开发流程
Landing Page
初级阶段
由运营和应用设计师/产品进行讨论，形成草图
由应用开发工程师用基于草图和DS，做出高保真的界面，收取反馈
根据反馈修改，直到获得最终确认
一键部署
高级阶段
用运营人员通过某种编辑器自己做出来，一键部署，根据用户反馈修改，不需要开发人员参与
中台系统应用
初级阶段
由运营人员提出需求，由应用设计师/产品进行讨论，形成草图
由应用开发工程师用基于草图和DS，做出高保真的界面，进行一次演示，收取反馈
根据反馈修改，直到获得最终确认
后端应用开发工程师基于界面、业务的理解和以往的系统知识，设计后端，并实现后端
再一次演示，收取反馈
根据反馈修改，直到获得最终确认
一键部署
高级阶段
运营人员自己给出草图
全栈应用开发人员先基于草图和DS做岀高保真的前端，进行一次演示，收取反馈
根据反馈修改，直到最终确认
该全栈应用开发人员再基于前端和自己对业务的理解和以往的系统知识，设计后端，并实现后端
再一次演示，收取反馈
根据反馈修改，直到获得最终确认

DS的维护流程

原则
以减少歧义为第一优先原则
不使用缩写
必须有明确的使用要求定义，不能模棱两可
比如BEM到底用”—“还是”_”怎么用要有明确的说法并附上例子，不能模棱两可，怎么写也行。
统一的convention，消灭同义词和相似的实现方法
通过木桶测试

命名要有层次

追求公式化、可组合，减少特例
尊从奥卡姆剃刀原理，如无必要勿增实体
