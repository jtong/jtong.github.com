已有前端代码资产与变化源的关系
为什么有本文？
在精益创业方法大行其道的今天，数字平台战略（http://gigix.thoughtworkers.org/2017/4/25/ditigal-platform-strategy-intro/）逐渐进入人们的视野。为实现数字平台战略，一个组织的响应力至关重要。所谓响应力就是指，每当组织响应市场反馈，进行一次“假说-实验”时，这个“假说-实验”的闭环周期越短，响应力就越强。
响应力的提升不是靠单一角色的变化就可以完成，需要各个相关角色之间的互相理解、配合才能做到。为了增进UX和Dev的互相了解，所以写了本文。
名词解释
变化源
是什么？
设想一个发布的软件，随着市场的反馈回来，我们总要进行一系列变化，这些变化站在代码资产的视角，会发现有下面几类（参见分类解释）。这每一类我们称之为一种变化源
为什么要站在代码资产的角度看待变化源？
在数字化时代，代码是一切业务价值实现的最终瓶颈，不管其他人的视角是如何的，如果开发人员不能理解，那么就直接影响响应力，而且开发人员大量的时间是在实现代码，去理解全局的时间有限，所以分类上照顾开发人员的视角是一个相对经济的做法。
分类解释
用户体验
指的是为了让用户体验好一点而做的改进，变化频率最高
业务逻辑
指的是界面背后的纯逻辑，变化频率一般
设计风格
指的是设计语言的选择，整个设计系统的搭建，如果干了这个工作的话，变化频率应该是最低的

http://karrisaarinen.com/dls/
可能的配合工具： https://github.com/airbnb/react-sketchapp
程序员随机起名字，搭架子
指的是程序员在不理解上述内容的内在逻辑的情况下，随便起的名字，塔的架子，变化频率随机，看人员能力
在代码资产中不会提及这个变化，因为理论上应该避免
然而实际上不能更多了
代码资产
是什么？
顾名思义是我们已经在服务器上运转，为组织创造价值的在线程序的代码。
分类方式
先是按语言分类
然后是按隔离变化源分类，也就是我们看到的分类法已经在试图隔离变化源了
如何看待下面的关系
当变化源是“设计风格”和“业务逻辑”时
建立统一语言
什么是统一语言
随便指定一个领域，以业务为例，那么领域里有业务名词，还有相关的业务动作（动词），等一系列词汇
还有诸如审计、法律、组织模式等可以用于组织词汇形成语句的各种模式，即所谓的逻辑关系
这一系列词汇和词汇关系组成的一种人们在某一领域里进行交流的语言，我们叫统一语言
好处是：
减少“程序员随机起名字，搭架子”的变化源
当然也减少其他角色之间的沟通混乱
由于统一语言的变化往往是比较慢的，所以在时间维度上，可以在一段时期减少变化源
当变化源是“用户体验”的时候，意味着剧烈的变化和沟通不畅
程序员采取的措施是：
组件化，在最外层做好封装
组件内不追求复用
组件内逻辑要简单，清晰
频繁展示结果
UX采取的措施是：
必须建立统一语言，防止程序员在不理解设计语言的情况下混淆设计风格和用户体验
频繁查看结果
